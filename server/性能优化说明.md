# HwMonServer 性能优化说明

## 问题描述

系统在高并发场景下出现卡顿和信息刷新不出来的问题,后台日志显示:
```
pymysql.err.OperationalError: (1205, 'Lock wait timeout exceeded, try restarting transaction')
```

## 根本原因

1. **数据库锁竞争**: 多个客户端同时上报数据时,对 `clients` 表进行 `INSERT ... ON DUPLICATE KEY UPDATE` 操作,造成行锁冲突
2. **事务过长**: 数据上报事务包含了太多操作(插入、查询、对比、发邮件等),导致锁被长时间持有
3. **读写混合使用同一连接池**: 只读查询和写操作混用,增加了不必要的锁竞争
4. **缺少重试机制**: 遇到锁超时直接失败,没有自动重试

## 优化方案

### 1. 分离读写连接池

**优化前**: 所有操作都使用 `get_db()`,包括只读查询
**优化后**: 
- `get_db()` - 用于写操作(INSERT/UPDATE/DELETE)
- `get_db_readonly()` - 用于只读查询(SELECT),设置 `READ ONLY` 事务模式

**优化的接口**:
- `/api/clients` - 获取客户端列表
- `/api/groups` - 获取分组列表
- `/api/export/csv` - 导出CSV
- `/api/export/json` - 导出JSON
- `/api/export/excel` - 导出Excel
- `/api/alerts` - 获取告警记录
- `/api/email-config` - 获取邮件配置
- `/api/alert-settings` - 获取告警设置
- `/api/collect/<client_id>` - 主动采集单个客户端(读取部分)
- `/api/collect/all` - 批量采集(读取部分)
- `send_alert_email()` - 发送邮件时读取配置

### 2. 优化事务处理

**优化前**: 整个数据上报流程在一个长事务中完成
**优化后**: 拆分为多个短事务

```python
# 步骤1: 快速更新客户端信息(单独事务,立即提交释放锁)
cursor.execute('INSERT INTO clients ... ON DUPLICATE KEY UPDATE ...')
conn.commit()  # 立即提交

# 步骤2: 保存硬件历史数据
cursor.execute('INSERT INTO hardware_history ...')
conn.commit()  # 立即提交

# 步骤3: 硬件变更检测(独立处理,不阻塞主流程)
_check_hardware_changes(...)
```

### 3. 添加重试机制

**数据上报接口**:
```python
max_retries = 3
for attempt in range(max_retries):
    try:
        return _process_report(attempt + 1, max_retries)
    except Exception as e:
        if 'Lock wait timeout' in str(e) and attempt < max_retries - 1:
            time.sleep(0.5 * (attempt + 1))  # 递增延迟重试
            continue
        else:
            raise e
```

**数据库连接获取**:
```python
def get_db():
    max_retries = 3
    for attempt in range(max_retries):
        try:
            conn = db_pool.connection()
            cursor.execute("SET time_zone='+08:00'")
            return conn
        except Exception as e:
            if attempt < max_retries - 1:
                time.sleep(0.5)
            else:
                raise e
```

### 4. 优化MySQL配置

**新增配置参数**:
```python
MYSQL_CONFIG = {
    'init_command': "SET time_zone='+08:00', innodb_lock_wait_timeout=10",
    'connect_timeout': 10,   # 连接超时10秒
    'read_timeout': 30,      # 读取超时30秒
    'write_timeout': 30      # 写入超时30秒
}
```

**锁等待超时**: 从默认的50秒降低到10秒,快速失败并触发重试

### 5. 增加连接池容量

**优化前**: `maxconnections=50`
**优化后**: `maxconnections=100`

支持更高并发场景下的连接需求

## 优化效果

### 预期改进

1. **减少锁等待时间**: 只读查询不再参与锁竞争,锁等待超时错误大幅减少
2. **提高响应速度**: 短事务快速提交,减少锁持有时间
3. **增强容错能力**: 自动重试机制处理偶发的锁冲突
4. **提升并发能力**: 连接池扩容支持更多并发请求

### 监控建议

部署后建议监控以下指标:
- 数据库锁等待超时错误频率
- API响应时间(P95/P99)
- 数据库连接池使用率
- CPU和内存使用率

## 进一步优化建议

如果优化后仍有性能问题,可以考虑:

1. **数据库层面**:
   - 为常用查询字段添加索引
   - 优化慢查询SQL
   - 考虑读写分离架构

2. **应用层面**:
   - 实现前端数据缓存(减少重复请求)
   - 使用WebSocket推送代替轮询
   - 异步处理硬件变更检测和邮件发送

3. **架构层面**:
   - 引入消息队列(RabbitMQ/Redis)缓冲高并发写入
   - 使用Redis缓存热点数据
   - 考虑微服务架构拆分
